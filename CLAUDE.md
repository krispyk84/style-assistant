# Vesture — Working Reference

> This file is the single source of truth. `AGENTS.md` is a symlink to it so Claude Code and Codex both load the same content.

## Project

Vesture is an AI-powered personal style assistant. Users build a digital closet, generate outfit recommendations for occasions and weather, plan trip wardrobes day-by-day, receive second opinions from AI stylists, and plan a 7-day outfit calendar. The backend generates recommendations via OpenAI and produces fashion sketches via Fal / Gemini / Imagen.

## Tech Stack

| Layer | Technology |
|---|---|
| Frontend | Expo SDK, React Native, TypeScript, expo-router (file-based routing) |
| Backend | Node.js, Express, Prisma, PostgreSQL |
| AI | OpenAI (outfit generation, closet analysis, second opinions, trip outfits), Fal / Gemini / Imagen (sketch generation) |
| Auth | Supabase Auth |
| DB | Supabase (PostgreSQL via Prisma) |
| Storage | Supabase Storage (images), AsyncStorage (local device state) |

## Deploy

- **Backend (production)**: `git push origin main` → GitHub Actions runs typecheck → on green, fires `RENDER_DEPLOY_HOOK_URL` → Render builds + deploys `style-assistant-api`.
- **Backend (dev)**: `git push origin dev` → GitHub Actions runs typecheck → on green, fires `RENDER_DEPLOY_HOOK_URL_DEV` → Render builds + deploys `style-assistant-api-dev`. Points at the **dev Supabase project** (separate from prod). Both services are defined in `render.yaml` as a Render Blueprint. **Render's auto-deploy is disabled** in the dashboard for both services — the GitHub Action is the sole deploy trigger so a failing typecheck never reaches Render.
- **CI**: `.github/workflows/deploy.yml` runs frontend `tsc --noEmit` and backend `prisma generate + tsc --noEmit` on every PR and on every push to `main` / `dev`. Deploys only fire after typecheck passes.
- **iOS (production)**: `npm run deploy` → `expo prebuild --clean -p ios` then xcodebuild Release to the connected iPhone, installs as **Vesture** (`com.krispyk84.styleassistant`).
- **iOS (dev side-by-side)**: `npm run deploy:dev` → same pipeline with `APP_VARIANT=development`, installs as **Vesture Dev** (`com.krispyk84.styleassistant.dev`) alongside the production app. By default the dev build points at the hosted dev backend (`https://style-assistant-api-dev.onrender.com`). Override with `DEV_API_URL=http://192.168.1.190:4000 npm run deploy:dev` to point at a backend running on your laptop instead (useful when iterating on backend changes without pushing to the `dev` branch).
- **Backend (local dev)**: `npm run backend:dev` (or `cd backend && npm run dev`) — Express + tsx watch on port 4000. Reachable from the iPhone over WiFi at the Mac's LAN IP. Note: backend secrets in `backend/.env` still point at production Supabase / OpenAI / Storage by default — local backend code, real downstream services.

### Dev variant config
`app.config.ts` swaps the bundle id, name, and URL scheme when `APP_VARIANT=development`:

| Field | Production | Development |
|---|---|---|
| `name` | Vesture | Vesture Dev |
| `ios.bundleIdentifier` | com.krispyk84.styleassistant | com.krispyk84.styleassistant.dev |
| `scheme` | styleassistant | styleassistantdev |
| Workspace generated by prebuild | `ios/Vesture.xcworkspace` | `ios/VestureDev.xcworkspace` |

Firebase + Apple Sign-In plugins are active in **both** variants. In the dev build, Firebase fails to initialize at runtime (the `.dev` bundle id doesn't match `GoogleService-Info.plist`'s configured bundle id) and logs a warning — accepted soft failure, no analytics in dev. Apple Sign-In may also fail at runtime until you register the `.dev` bundle id with the capability in the Apple Developer portal (`-allowProvisioningUpdates` in the build script tries to do this automatically).

`ios/` is gitignored — every deploy regenerates it via `expo prebuild --clean`. Both `deploy` and `deploy:dev` run prebuild, so the local `ios/` is always in the variant matching whichever script you ran last. Don't expect `ios/` to be a stable working tree.

The `extra.appVariant` value (`development` | `production`) is exposed via `expo-constants` for runtime labelling if you ever want a "DEV" badge on screen.

### plugins/with-modular-headers.js
Tiny custom Expo config plugin that injects `use_modular_headers!` into the generated Podfile. Required because @react-native-firebase pulls in Swift pods that depend on Objective-C pods without module maps — pod install fails as a static-library build without modular headers. The Firebase Expo plugin doesn't inject this itself. Plugin runs at every prebuild so the fix is automatic for both prod and dev.

### plugins/with-development-team.js
Tiny custom Expo config plugin that writes `DEVELOPMENT_TEAM` into every Xcode build configuration. Required because `expo prebuild` generates a fresh `project.pbxproj` with no team set, and xcodebuild fails with "Signing for ... requires a development team." Default team is hardcoded; override per-machine with `APPLE_TEAM_ID=...` in the environment.

## Key Conventions

### Modularization pattern
Every complex screen or component is split into three layers, created in this order:
1. **Hook(s)** (`use*.ts`) — all state, effects, and logic; no JSX
2. **View** (`*Screen.tsx` / `*View.tsx`) — pure UI; calls hooks and renders
3. **Entry** (`*.tsx` at the Expo Router path) — single re-export line

Entry files exist only to satisfy Expo Router's file-based routing. Never put logic in them.

A screen is considered **complex enough to split** when it has multiple pieces of state, async effects, or non-trivial handlers. Screens that are essentially declarative (one or two state slots, no async work, JSX wrapping a single sub-component) are left as single files — the pattern serves the code, not the other way around.

### Cross-hook coordination
When hook A needs to update state owned by hook B, the screen assembles a coordinator function. Hook B passes its state setter out; the screen wires them together. Hooks never import each other.

### State setter params
Hooks that perform mutations receive state setters as constructor params:
```ts
useWeekPlanActions({ setItems, setSavedOutfitIds, savedOutfitIds })
useClosetItemSubmit({ item, setError, setIsEditing, onSaved, onDeleted })
```

### getFields() snapshot pattern
Form hooks expose a `getFields(): FieldType` function that snapshots all field values at call time. The submit hook calls `handleSave(editor.getFields())` rather than holding a reference to the editor hook.

### useFocusEffect
Always use `useFocusEffect(useCallback(...))`, never `useEffect([isFocused])`. The latter fires on both focus gain and loss, causing concurrent hydration calls with racing isMounted closures.

### Rules of Hooks
All hook calls must appear before any conditional return. If a screen has an early return (`if (!data) return <ErrorState />`), every hook still executes above it.

### Mapper pattern
Field values → API request shapes are handled in dedicated `*-mappers.ts` files, not inline in submit hooks or form components.

### Service layer
Each domain has: `*-service.ts` (interface), `api-*-service.ts` (real impl), `mock-*-service.ts` (mock), `index.ts` (exports the active impl based on env flag).

### Static theme
`theme` is imported directly from `@/constants/theme` (static object). `useTheme()` is used only when the component needs to respond to appearance mode changes at runtime.

### Route builders
Trip nav uses `lib/trip-route.ts` builders (`buildTripAnchorsHref`, `buildTripResultsHref`, etc.) rather than inline `router.push({ pathname: '/...' })` strings. Look-flow nav uses `lib/look-route.ts` builders. Add new routes to the corresponding builder file rather than scattering pathname literals.

### Shared sketch panel
All ready/pending/loading/failed sketch UI goes through `components/generated/GeneratedSketchPanel`. Cards configure it with `mode='large' | 'compact'` and per-context labels. Do not reimplement the four-state branch inline.

### Shared outfit-piece list
Look results, trip days, and the tier-detail "check recommended pieces" list all render through `components/cards/OutfitPieceListView`:
- `display='labeled'` for the look-results tier card (label/value rows).
- `display='grouped'` for trip days (category-grouped bullets).
- `display='card'` for tier-detail rows (full-row Pressable with optional `trailingIcon`, e.g. a camera icon to enter the check-piece flow).

The component owns the matched-item bottom sheet (`ClosetItemSheet`) and per-suggestion thumbs feedback wiring; pass `onMatchThumbsUp` / `onMatchThumbsDown` to enable rematch, omit them for view-only. All three modes feed the same `LabeledPiece[]` shape — built via `buildLabeledPieces` (look results), `buildPiecesToCheck` (tier detail), or `buildTripDayLabeledPieces` (trip days); the type lives in `lib/outfit-piece-display.ts`. Do not reimplement piece rows, the "you already own a similar piece" hint, or per-piece match resolution inline.

### Trip prompt rules
`backend/src/ai/prompts/trip-shared-rules.ts` is the single source of truth for the bag-type taxonomy and temperature-rule lines used by both trip generation and per-day regeneration. Do not duplicate these constants inline.

## Notes

### Screens that intentionally aren't split
- `app/review-request.tsx` — entry-only. The screen parses URL params via `parseLookInput`, has two early-return guards, and one nav handler. No state, no effects worth a hook.
- `app/(app)/profile.tsx` — entry-only. One `isEditing` boolean and one save handler that delegates to `useAppSession.saveProfile`. The form's logic lives in `useProfileForm` / `useProfileFormSubmit`.
- `app/create-look.tsx` — entry-only. The form is split via `CreateLookRequestForm` → container/view/hooks. The screen reads optional closet-item route params via `buildAnchorItemsFromClosetParams` and renders the form.

### useLogout.ts
Currently a one-line wrapper around `signOut`. Intended as the placeholder for logout cleanup (AsyncStorage clear, cache wipe, analytics flush). Needs real content when logout flow expands.
